ACCEPTANCE CRITERIA
-------------------
By hour 16, a demo should be able to:

1. Start backend + frontend locally
2. Connect in demo mode (no wallet required)
3. Request a loan -> see real (or mock) loan ID
4. View loan details
5. Vote as oracle 1 + 2 -> see consensus reached
6. Trigger slashing -> see loan marked as defaulted

The smart contract may still be mocks behind the scenes
if real compilation fails, but the demo flow must work end-to-end.


FALLBACK PLAN (if contract compilation fails)
----------------------------------------------
If Compact compilation / testnet deployment is too complex:
- Keep current mock circuit layer but make it more convincing
- Add realistic timing delays to simulate proof generation
- Add better mock data that looks like real contract state
- Polish the demo narrative: "This shows what the real contract does"
- Focus on UX polish and demo script quality instead
